# Стратегия и оркестрация

Данный каталог содержит полную стратегическую документацию системы **NEXUS** (Network of EXperts, Unified in Strategy) — операционную доктрину для многоагентной оркестрации в рамках The Agency. NEXUS превращает независимых ИИ-специалистов из 9 подразделений (инженерия, дизайн, маркетинг, продукт, управление проектами, тестирование, поддержка, пространственные вычисления и специализированные операции) в синхронизированную интеллектуальную сеть с определёнными конвейерами, контрольными точками качества и измеримыми результатами.

## Корневые файлы

### Краткое описание для руководства (`EXECUTIVE-BRIEF.md`)

Исполнительная справка по системе NEXUS, предназначенная для лиц, принимающих решения. Документ содержит обзор ситуации, описывающий проблему координации между 9 подразделениями ИИ-агентов. Приводятся четыре ключевых вывода: 73% многоагентных проектов терпят неудачу на этапе передачи задач без структурированных протоколов; необходимость подхода «по умолчанию — требуется доработка» для предотвращения преждевременного развёртывания; параллельное выполнение по 4 трекам сокращает сроки на 40–60%; цикл «Разработка-Тестирование» с максимумом 3 попыток выявляет 95% дефектов до интеграции. Описаны три режима развёртывания (NEXUS-Full на 12–24 недели, NEXUS-Sprint на 2–6 недель, NEXUS-Micro на 1–5 дней), бизнес-эффект, рекомендации с приоритетами и конкретные следующие шаги.

### Руководство быстрого старта (`QUICKSTART.md`)

Пятиминутное руководство по запуску оркестрированного многоагентного конвейера. Содержит готовые к копированию промпты для каждого из трёх режимов NEXUS: Full (полный проект с нуля), Sprint (фича или MVP) и Micro (конкретная задача). Для режима Micro приведены шаблоны для шести типовых сценариев: исправление бага, запуск маркетинговой кампании, комплаенс-аудит, расследование проблем производительности, маркетинговое исследование и улучшение UX. Также содержит справочную таблицу всех стратегических документов, краткое описание ключевых концепций NEXUS (контрольные точки качества, цикл Разработка-Тестирование, передача контекста, Reality Checker, доказательства вместо утверждений) и визуальную карту всех агентов по подразделениям.

### Основная стратегия NEXUS (`nexus-strategy.md`)

Полная операционная доктрина системы NEXUS объёмом более 800 строк. Это центральный документ, определяющий всю модель оркестрации. Включает: стратегическое обоснование и основные принципы (целостность конвейера, непрерывность контекста, параллельное выполнение, доказательная база, быстрый провал и исправление); полный реестр агентов по 9 подразделениям с описанием их ролей в конвейере; семифазную модель конвейера (от разведки и обнаружения до эксплуатации и развития); матрицу координации агентов, определяющую, кто с кем взаимодействует; протоколы передачи задач между агентами; систему контрольных точек качества для каждой фазы; управление рисками с матрицей эскалации; метрики успеха по категориям (надёжность, рост, удержание, финансы, качество, эффективность); и краткое руководство по активации.

## Каталог `coordination/` — Координация

### Промпты активации агентов (`coordination/agent-activation-prompts.md`)

Готовые к использованию шаблоны промптов для активации любого агента в конвейере NEXUS. Документ организован по подразделениям и содержит детализированные промпты с заполняемыми полями для следующих агентов и ролей: Agents Orchestrator (полный конвейер и цикл Разработка-Тестирование), инженерное подразделение (Frontend Developer, Backend Architect, AI Engineer, DevOps Automator, Rapid Prototyper), дизайн-подразделение (UX Architect, Brand Guardian), подразделение тестирования (Evidence Collector для пошагового QA, Reality Checker для финальной интеграционной проверки, API Tester), продуктовое подразделение (Sprint Prioritizer) и подразделение поддержки (Executive Summary Generator). Каждый промпт содержит конкретные требования к результатам, ссылки на входные документы, критерии приёмки и указания на следующего агента в цепочке. Также приведена справочная таблица, какой промпт использовать в какой ситуации.

### Шаблоны передачи задач (`coordination/handoff-templates.md`)

Стандартизированные шаблоны для всех типов передачи задач между агентами в конвейере NEXUS. Содержит семь шаблонов: (1) Стандартная передача — универсальный формат для любой передачи работы от агента к агенту с метаданными, контекстом, требованиями к результату и ожиданиями по качеству; (2) QA-вердикт «Принято» — используется когда Evidence Collector или другой QA-агент утверждает задачу, включает доказательства в виде скриншотов и функциональную верификацию; (3) QA-вердикт «Отклонено» — детализированный отчёт о найденных проблемах с описанием ожидаемого и фактического поведения, инструкциями по исправлению и указанием конкретных файлов; (4) Отчёт об эскалации — используется когда задача не прошла 3 попытки, включает историю неудач, анализ коренных причин и рекомендации по разрешению; (5) Межфазовая передача — для перехода между фазами NEXUS с результатами контрольных точек и активацией агентов следующей фазы; (6) Спринтовая передача — для передачи контекста между спринтами с метриками качества и ретроспективой; (7) Инцидентная передача — для передачи контекста во время реагирования на инциденты с хронологией, текущим состоянием и информацией для следующего респондента.

## Каталог `playbooks/` — Плейбуки фаз

### Фаза 0: Разведка и обнаружение (`playbooks/phase-0-discovery.md`)

Плейбук фазы разведки продолжительностью 3–7 дней с участием 6 агентов. Цель — валидация возможности до выделения ресурсов. Агенты запускаются двумя параллельными волнами: первая волна включает Trend Researcher (анализ рынка, конкурентный ландшафт, прогноз трендов), Feedback Synthesizer (анализ потребностей пользователей, многоканальный сбор обратной связи, оценка болевых точек по RICE) и UX Researcher (поведенческий анализ, персоны, карты пользовательских путей); вторая волна — Analytics Reporter (аудит данных и источников сигналов), Legal Compliance Checker (регуляторный скан, оценка рисков соответствия) и Tool Evaluator (оценка технологического ландшафта, анализ build vs buy). В точке конвергенции Executive Summary Generator синтезирует все отчёты в исполнительное резюме с рекомендацией GO/NO-GO/PIVOT. Контрольная точка качества проверяет валидацию рыночной возможности, наличие подтверждённых болевых точек, отсутствие блокирующих проблем соответствия и готовность технологического стека.

### Фаза 1: Стратегия и архитектура (`playbooks/phase-1-strategy.md`)

Плейбук стратегической фазы продолжительностью 5–10 дней с участием 8 агентов. Цель — определить, что строится, как это структурировано и как выглядит успех, до написания единой строки кода. Выполняется в три последовательных шага: (1) Стратегическое обрамление — Studio Producer (стратегическое портфельное выравнивание, ROI-цели), Brand Guardian (система бренд-идентичности: цвета, типографика, голос бренда), Finance Tracker (бюджет, модель ROI, анализ безубыточности); (2) Техническая архитектура — UX Architect (CSS дизайн-система, компонентная архитектура, система тем), Backend Architect (архитектура системы, схема БД, проектирование API, архитектура безопасности), AI Engineer (архитектура ML-системы при необходимости), Senior Project Manager (преобразование спецификации в задачи с критериями приёмки); (3) Приоритизация — Sprint Prioritizer (бэклог с RICE-скорами, спринт-план, MoSCoW-классификация). Требуется двойное утверждение: Studio Producer (стратегия) и Reality Checker (техника).

### Фаза 2: Фундамент и каркас (`playbooks/phase-2-foundation.md`)

Плейбук фазы создания фундамента продолжительностью 3–5 дней с участием 6 агентов. Цель — построить техническую и операционную основу, от которой зависит вся последующая работа. Два параллельных потока: Поток A (Инфраструктура) — DevOps Automator (CI/CD-конвейер с этапами безопасности, тестирования, сборки и развёртывания; инфраструктура как код; управление секретами), Infrastructure Maintainer (облачные ресурсы, стек мониторинга Prometheus/Grafana, логирование, усиление безопасности), Studio Operations (Git-workflow, каналы коммуникации, шаблоны документации); Поток B (Приложение) — Frontend Developer (каркас проекта, библиотека компонентов, оболочка приложения с маршрутизацией), Backend Architect (база данных с миграциями, API-каркас с аутентификацией, middleware), UX Architect (реализация дизайн-токенов, система лэйаутов, система тем свет/тёмный). Evidence Collector верифицирует все элементы фундамента скриншотами.

### Фаза 3: Разработка и итерации (`playbooks/phase-3-build.md`)

Плейбук основной фазы разработки продолжительностью 2–12 недель с участием 15–30+ агентов. Это центральная фаза, где NEXUS-оркестрация приносит наибольшую ценность. Основной механизм — цикл «Разработка-Тестирование» (Dev-QA loop): для каждой задачи из бэклога по RICE-приоритету выполняется назначение разработчику, реализация, тестирование Evidence Collector, и решение PASS/FAIL с максимумом 3 попыток до эскалации. Приведена подробная матрица назначения агентов: 14 категорий задач от React UI до CLI-инструментов, каждая с основным агентом, резервным агентом и QA-агентом. Для полного режима NEXUS-Full определены 4 параллельных трека: Core Product Development, Growth & Marketing Preparation, Quality & Operations, Brand & Experience Polish. Описан шаблон спринта (планирование, ежедневное исполнение, обзор спринта, ретроспектива) и логика принятия решений оркестратором при неудачах и параллельном управлении задачами.

### Фаза 4: Качество и закалка (`playbooks/phase-4-hardening.md`)

Плейбук финальной проверки качества продолжительностью 3–7 дней с участием 8 агентов. Единоличный контролёр — Reality Checker, чей вердикт по умолчанию — «ТРЕБУЕТСЯ ДОРАБОТКА». Выполняется в три шага: (1) Сбор доказательств — Evidence Collector (полный пакет скриншотов по всем устройствам, всем страницам, всем темам, всем состояниям ошибок), API Tester (полная регрессия API включая edge-кейсы), Performance Benchmarker (нагрузочное тестирование при 10x трафике, Core Web Vitals, стресс-тестирование), Legal Compliance Checker (финальный аудит приватности, безопасности, доступности); (2) Анализ — Test Results Analyzer (агрегация метрик качества, приоритизация проблем), Workflow Optimizer (обзор эффективности процессов), Infrastructure Maintainer (проверка готовности продакшена, аварийное восстановление); (3) Финальный вердикт — Reality Checker проводит обязательные проверки: верификация реальности, кросс-валидация QA, сквозная валидация пользовательских сценариев, поточечное сравнение со спецификацией. Три возможных вердикта: READY (редко с первого раза), NEEDS WORK (ожидаемый результат, возврат в фазу 3), NOT READY (фундаментальные проблемы, возврат в фазы 1/2).

### Фаза 5: Запуск и рост (`playbooks/phase-5-launch.md`)

Плейбук фазы запуска продолжительностью 2–4 недели (T-7 по T+14) с участием 12 агентов. Координация выхода на рынок по всем каналам одновременно. Хронология: T-7 (неделя до запуска) — подготовка контента и кампаний (Content Creator, Social Media Strategist, Growth Hacker, App Store Optimizer), техническая подготовка (DevOps Automator — blue-green deployment, Infrastructure Maintainer — автомасштабирование на 10x трафике, Project Shepherd — чек-лист запуска); T-1 — финальный чек-лист по техническим, контентным, маркетинговым и поддерживающим аспектам; T-0 (день запуска) — развёртывание в продакшн, активация всех маркетинговых каналов (Twitter, Reddit, Instagram, TikTok), мониторинг и реагирование (Support Responder, Analytics Reporter, Feedback Synthesizer); T+1 по T+7 — ежедневный ритм с утренней аналитикой, дневной оптимизацией контента и вечерними брифингами для стейкхолдеров; T+7 по T+14 — оптимизация каналов, когортный анализ, A/B-тесты, планирование итерационного спринта.

### Фаза 6: Эксплуатация и развитие (`playbooks/phase-6-operate.md`)

Плейбук бессрочной операционной фазы с участием 12+ агентов (ротация). Продукт запущен — теперь нужно обеспечить его процветание. Определены операционные ритмы на пяти уровнях: непрерывно (Infrastructure Maintainer — 99.9% аптайм, Support Responder — время первого ответа < 4ч, DevOps Automator — множественные деплои в день); ежедневно (Analytics Reporter — KPI-дашборд, мониторинг здоровья системы); еженедельно (аналитика, синтез обратной связи, планирование спринтов, оптимизация каналов роста); ежемесячно (исполнительные отчёты, финансовый обзор, мониторинг комплаенса, рыночная разведка, аудит бренда); ежеквартально (стратегический обзор портфеля, аудит эффективности процессов, регрессионное тестирование производительности, оценка технического долга). Описан цикл непрерывного улучшения «Измерение-Анализ-Планирование-Разработка-Валидация-Развёртывание», протокол реагирования на инциденты с 4 уровнями серьёзности (P0–P3), операции роста с ежемесячным обзором каналов, финансовые операции с отслеживанием MRR/ARR и unit economics, операции комплаенса и ежеквартальный стратегический обзор. Определены 13 метрик успеха по 6 категориям с целевыми значениями.

## Каталог `runbooks/` — Ранбуки сценариев

### Сценарий: Стартап MVP (`runbooks/scenario-startup-mvp.md`)

Ранбук для построения MVP стартапа в режиме NEXUS-Sprint за 4–6 недель с участием 18–22 агентов. Сценарий: новый продукт, который должен быстро валидировать соответствие продукта рынку. Понедельный план: Неделя 1 — сжатое обнаружение + архитектура (конкурентный скан за 1 день, вайрфреймы, список задач, CSS дизайн-система, архитектура системы, бренд-основа, RICE-бэклог, настройка CI/CD и каркас проекта); Недели 2–3 — основная разработка через цикл Dev-QA (ядро UI, ядро API, ML-фичи при необходимости, проектирование вирусных механик, создание контента запуска, настройка аналитики); Неделя 4 — полировка и закалка (полный пакет скриншотов, нагрузочное тестирование, финальная проверка Reality Checker с ожидаемым циклом доработки); Недели 5–6 — запуск и оптимизация каналов роста. Описаны ключевые точки принятия решений, критерии успеха (время до живого продукта, первые пользователи за 48 часов, 50+ отзывов за 2 недели) и типичные ловушки с мерами противодействия (расползание объёма, избыточная инженерия, пропуск QA ради скорости).

### Сценарий: Корпоративная фича (`runbooks/scenario-enterprise-feature.md`)

Ранбук для разработки крупной функции существующего корпоративного продукта в режиме NEXUS-Sprint за 6–12 недель с участием 20–30 агентов. Сценарий: добавление масштабной функциональности с жёсткими требованиями к комплаенсу, безопасности и качеству; необходимо согласование множества стейкхолдеров и бесшовная интеграция с существующими системами. Команда разделена на три группы: основная команда (15 агентов — от оркестратора до бенчмарка производительности), комплаенс и управление (4 агента — Legal Compliance Checker, Brand Guardian, Finance Tracker, Executive Summary Generator) и обеспечение качества (3 агента — Test Results Analyzer, Workflow Optimizer, Experiment Tracker). План по неделям: 1–2 — требования и архитектура с обзором стейкхолдеров; 3 — фундамент с feature-флагами; 4–9 — разработка в 3 спринтах с двухнедельными брифингами стейкхолдеров; 10–11 — закалка с нагрузочным тестированием при 10x трафике и финальным вердиктом Reality Checker; 12 — канареечное развёртывание (5% - 25% - 100%). Определены требования к качеству (покрытие кода > 80%, P95 API < 200ms, WCAG 2.1 AA, zero critical vulnerabilities) и матрица рисков.

### Сценарий: Многоканальная маркетинговая кампания (`runbooks/scenario-marketing-campaign.md`)

Ранбук для координированной маркетинговой кампании в режимах NEXUS-Micro или NEXUS-Sprint за 2–4 недели с участием 10–15 агентов. Три группы агентов: ядро кампании (Social Media Strategist как лидер, Content Creator, Growth Hacker, Brand Guardian, Analytics Reporter), платформенные специалисты (Twitter Engager, TikTok Strategist, Instagram Curator, Reddit Community Builder, App Store Optimizer) и поддержка (Trend Researcher, Experiment Tracker, Executive Summary Generator, Legal Compliance Checker). Неделя 1 — стратегия кампании (цели, KPI, аудитория, бюджет, контент-календарь, анализ трендов, дизайн воронки, комплаенс-ревью) и создание контента по всем платформам с учётом специфики каждой; Неделя 2 — запуск и активация всех каналов, оптимизация на основе данных первых 48 часов; Недели 3–4 — устойчивое ведение и оптимизация с ежедневной и еженедельной ритмикой. Определены KPI кампании (охват, вовлечённость > 3%, CTR > 2%, конверсия > 5%, CPA) и платформо-специфичные KPI для каждого канала. Контрольные точки согласования с брендом на каждом этапе.

### Сценарий: Реагирование на инциденты (`runbooks/scenario-incident-response.md`)

Ранбук для обработки производственных инцидентов в режиме NEXUS-Micro продолжительностью от минут до часов с участием 3–8 агентов. Определена четырёхуровневая классификация серьёзности: P0 (критический — сервис полностью недоступен, потеря данных, нарушение безопасности — немедленная реакция), P1 (высокий — основная функция сломана, значительная деградация — менее 1 часа), P2 (средний — второстепенная функция, есть обходной путь — менее 4 часов), P3 (низкий — косметическая проблема — следующий спринт). Для каждого уровня определён состав команды реагирования. Последовательность реагирования из 5 шагов: (1) Обнаружение и сортировка (0–5 минут) — Infrastructure Maintainer классифицирует и активирует команду; (2) Расследование (5–30 минут) — параллельная проверка метрик, логов, деплоев, БД и внешних зависимостей; (3) Смягчение (15–60 минут) — дерево решений в зависимости от причины (откат, масштабирование, хотфикс, активация фоллбэка); (4) Верификация решения — Evidence Collector и API Tester подтверждают исправление; (5) Пост-мортем (в течение 48 часов) — Workflow Optimizer проводит анализ коренных причин, оценку воздействия, определение мер предотвращения и формирование задач для бэклога. Включены шаблоны коммуникации для страницы статуса и исполнительных обновлений, а также матрица эскалации.
